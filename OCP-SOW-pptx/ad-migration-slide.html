Active Directory Migration - Execution Approach

🧩

Why Lift & Shift Is Not an Option

No-Go for direct DC move
  • Active Directory is replication-sensitive; snapshot-style VM relocation increases risk of USN rollback, replication divergence, and trust instability.
  • Domain controllers are tightly coupled to DNS, time hierarchy, and site topology. Moving existing VMs as-is can introduce authentication and policy delays.
  • Legacy VMware guest artifacts, inherited drift, and unknown hardening posture are carried forward, reducing migration confidence.
  • Rollback complexity is high for identity services, and blast radius is larger if existing DCs fail post-move.

Recommended OCP-V Migration Sequence

Go with build-and-promote
Execution Plan
  • Build new Windows Server VMs directly in OpenShift Virtualization and apply baseline hardening and patch compliance.
  • Join the new VMs to the existing domain, then promote them to domain controllers in the target OCP-V environment.
  • Validate AD health (replication, SYSVOL, DNS, authentication, GPO) and transfer FSMO roles to the new OCP-V DCs.
  • After production stabilization, demote legacy VMware DCs and remove them from service in a controlled decommission phase.
Planning principle: Active Directory migration should follow a build-and-promote model in OCP-V, then controlled FSMO transfer, then demotion and removal of legacy VMware DCs.
